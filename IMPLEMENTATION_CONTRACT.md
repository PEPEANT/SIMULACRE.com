# Implementation Contract

## 0. 목적

이 문서는 시뮬라크의 목표 시스템과 현재 구현 상태를 구분해서 설명하기 위한 계약 문서다.
핵심은 미래 목표를 정본으로 유지하되, 아직 구현되지 않은 것을 이미 구현된 것처럼 말하지 않게 만드는 것이다.

## 1. 상태 표기 규칙

- `implemented`: 현재 기본 동작이 존재하고 반복 사용 가능하다.
- `partial`: 일부만 존재하거나 조건부로만 동작한다.
- `planned`: 정본 기준에는 있으나 아직 구현되지 않았다.

문서를 쓸 때 `planned`를 `implemented`처럼 서술하지 않는다.

## 2. 초기 제품 우선순위

- 1차 본체는 3D UGC 오브젝트 엔진이다.
- 초기 코어 루프는 `배치 -> 저장 -> 전시`다.
- 초기 우선순위는 생성, 이동, 회전, 복제, 삭제, 저장, 불러오기다.
- 미니게임, 전투, 이벤트는 확장 모듈이며 초기 본체보다 앞서지 않는다.

## 3. 핵심 도메인 모델

### 3-1. 작품 모델

- 저장 기본 단위는 오브젝트가 아니라 작품(`creation/work`)이다.
- 작품 최소 필드는 아래를 가진다.
  - `owner`
  - `version`
  - `status`
  - `objects[]`
- 권장 확장 필드:
  - `creationId`
  - `createdAt`
  - `updatedAt`
  - `promotionHistory[]`
  - `exhibitTargets[]`

### 3-2. 상태 모델

- `draft`: 작업실 내부 초안
- `experimental`: 실험 월드 공개 상태
- `approved`: 메인 월드 승격 승인 상태
- `live`: 현재 전시/정착 상태
- `archived`: 보관 상태

### 3-3. 계층 모델

- `작업실`: 생성과 수정
- `실험 월드`: 검증과 피드백
- `메인 월드`: 승인된 창작물의 장기 정착

메인 월드 반영은 자동이 아니라 승격 절차를 거친다.

### 3-4. 객체 모델

- 오브젝트는 월드 재료다.
- 엔티티는 행동 가능한 존재다.
- AI는 엔티티에 부착되는 모듈이다.
- 이 세 계층을 하나의 타입처럼 취급하지 않는다.

## 4. 구현 원칙

- 현재 구현과 목표 계약을 항상 분리해서 서술한다.
- 임시 우회 동작은 장기 정책으로 승격하지 않는다.
- 저장 데이터는 버전 필드를 통해 마이그레이션 가능해야 한다.
- 포털 이동, 작품 승격, 전시는 모두 연속성 계약을 깨지 않는 방향으로 구현한다.

## 5. 근시기 구현 우선순위

1. 오브젝트 엔진 안정화
2. 작품 단위 저장/불러오기
3. 전시와 월드 갤러리 연결
4. 포털 이동 시 플레이어 연속성 복구
5. 작업실 -> 실험 -> 메인 승격 절차 구현
